# 자동 로그인 지원 기술 설명

현대적인 모바일 앱이나 웹 서비스에서 매번 앱을 켤 때마다 아이디와 비밀번호를 입력하는 번거로움을 없애기 위해 **자동 로그인** 기능을 필수로 지원합니다. Polbook 서비스에 자동 로그인을 구현하기 위한 대표적인 기술 원리와 방법을 설명합니다.

---

## 1. Refresh Token을 활용한 자동 로그인 (가장 권장됨)

JWT(JSON Web Token) 로그인을 사용할 때 보안과 편의성을 모두 잡기 위해 사용하는 가장 일반적인 방식입니다. 단순히 수명이 긴 토큰 하나만 쓰는 것이 아니라, **짧은 수명의 토큰**과 **긴 수명의 토큰** 두 가지를 운용합니다.

1.  **Access Token (액세스 토큰):**
    *   실제 서버에 요청을 보낼 때 사용하는 출입증입니다.
    *   보안을 위해 수명을 매우 짧게 (예: 30분 ~ 1시간) 설정합니다.
    *   탈취당하더라도 짧은 시간 안에 만료되므로 피해를 최소화할 수 있습니다.
2.  **Refresh Token (리프레시 토큰):**
    *   액세스 토큰이 만료되었을 때, 이를 **재발급받기 위한 용도**의 특별한 교환권입니다.
    *   수명을 길게 (예: 2주 ~ 1달) 설정합니다.
    *   클라이언트(앱/웹) 내부의 안전한 저장소에만 보관하며, 평소 API 요청에는 사용하지 않습니다.

### 자동 로그인 플로우
1.  사용자가 처음 로그인(ID/PW 입력)에 성공하면, 서버는 Access Token과 Refresh Token 두 개를 모두 발급하여 클라이언트에게 줍니다.
2.  클라이언트는 이 두 토큰을 휴대폰 로컬 저장소(Secure Storage)나 웹 브라우저의 안전한 공간(HttpOnly Cookie 등)에 저장합니다.
3.  사용자가 앱을 종료했다가 몇 시간, 며칠 뒤 다시 켭니다.
4.  앱은 서버에 저장해둔 Access Token을 실어 화면 보호용 API(예: `/me` 내 정보 가져오기)를 요청합니다.
5.  **만약 Access Token이 1시간이 지나 만료되었다면?**
    *   서버는 "401 Unauthorized (토큰 만료됨)" 에러를 응답합니다.
    *   앱(클라이언트 로직)은 당황하지 않고, 뒤에서 조용히 보관하고 있던 **Refresh Token**을 서버의 특정 재발급 엔드포인트(예: `/auth/refresh`)로 보냅니다.
    *   서버는 Refresh Token이 유효한지(DB 대조 등) 확인하고, 유효하다면 **새로운 Access Token**을 발급하여 내려줍니다.
    *   앱은 새로 받은 Access Token으로 처음에 실패했던 요청을 다시 보내어 성공적으로 화면을 그립니다.
6.  **사용자 입장:** 뒷단에서 이런 토큰 갱신 과정이 순식간에 일어나므로, 앱을 켜자마자 로그인된 화면을 바로 볼 수 있습니다. (자동 로그인 성공!)

## 2. 기타 보조 수단

위의 Refresh Token 방식이 베이스가 되지만, 이를 돕거나 다른 방식으로 자동 로그인을 구현하기도 합니다.

*   **웹 (Web) - 쿠키(Cookie) 기반의 자동 로그인 / "Remember Me":**
    *   전통적인 세션 기반에서 많이 쓰던 방식입니다.
    *   로그인 시 "로그인 상태 유지" 체크박스를 선택하면, 서버가 아주 수명이 긴 특별한 난수 형태의 토큰(Remember-me token)을 쿠키에 담아줍니다.
    *   다음 접속 시 세션이 만료되었더라도, 이 특별한 쿠키가 남아있으면 서버가 확인하여 세션을 자동으로 다시 만들어줍니다.
*   **모바일 (앱) - 생체 인증 (Biometrics) 연동:**
    *   토큰이 만료되어 재로그인이 필요할 때, 아이디/패스워드를 치라고 하는 대신 기기에 저장된 지문이나 Face ID를 요구합니다.
    *   생체 인증에 성공하면 OS 내부에 안전하게 잠가둔 ID/PW나 마스터 토큰을 꺼내어 서버에 자동으로 다시 로그인 요청을 보내는 방식입니다. (은행 앱에서 자주 사용됨)

---

### 결론 요약
Polbook은 JWT 인증 구조를 채택할 예정이므로, **Access Token + Refresh Token 구조**를 이용하여 보안성을 챙기면서도 앱/웹을 켤 때 끊김 없는 자동 로그인 경험을 제공할 수 있도록 설계하는 것이 좋습니다.
