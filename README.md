# 📚 Polbook (교내 중고 책 거래 서비스) 기획안

## 1. 서비스 개요
- **목적:** 교내 학생들 간의 안전하고 편리한 전공/교양/자격증 서적 중고 거래 지원
- **타겟 유저:** `.kopo.ac.kr` 이메일을 소유한 교내 재학생 및 교직원

## 2. 핵심 기능 요구사항 및 세부 로직

### 🔑 2.1 회원가입 및 로그인 (인증/인가)
- **회원가입:**
  - `학번@office.kopo.ac.kr` 웹메일을 통한 이메일 인증 필수 (외부인 차단)
  - 인증 완료 후 닉네임, 비밀번호 등을 설정하여 가입 완료
- **로그인:**
  - **ID:** 학번
  - **PW:** 사용자 지정 비밀번호 
  - **인증 방식:** 토큰 기반(JWT) 인증 및 자동 로그인 지원. (JWT는 서버 메모리에 세션을 유지할 필요가 없어 부하가 적고, 모바일 앱과 웹 양쪽 모두에서 동일한 방식으로 손쉽게 연동할 수 있는 훌륭한 확장성을 가집니다.)

### 📖 2.2 도서 등록 및 카테고리 분류
- **카테고리 구분:**
  1. `전공 서적` (전공 서적 선택시에만 '학과/전공'을 옵션이나 해시태그로 입력받습니다. 교양/자격증 서적은 특정 학과에 종속되지 않으므로, 이를 공통 필드로 빼지 않고 옵션으로 두어 깔끔한 데이터 분리를 위함입니다.)
  2. `교양 서적` 
  3. `자격증 서적`
- **게시글 필드:** 
  - 기본 정보: 책 제목, 판매 가격, 상세 설명, 사진 첨부
  - 상세 정보(선택): 학년, 학기, 강의명, 교수명
  - 상태 정보: 책 제본/상태(S, A, B급 등), 필기 여부

### 📍 2.3 교내 전용 거래 장소 지정
- 텍스트로 자유롭게 입력하는 대신, **사전에 정의된 교내 건물 입구**를 드롭다운으로 선택하도록 제한.
  - *지정 장소 목록 (건물 입구):* 본관, 1기술관, 2기술관, 3기술관, 백두관, 5기술관, 6기술관, 7기술관, 하이테크관, 한라관, 학생회관, 산학협력관
- 안전한 장소에서의 거래를 유도하고 동선을 최소화.

### 🔄 2.4 거래 상태 관리 및 안전 결제 (에스크로)
게시글 내에서 판매자가 직접 거래 상태를 변경할 수 있도록 구성합니다.
- `판매중` : 현재 구매 가능한 상태
- `예약중` : 구매자가 **안전 결제(에스크로)**를 통해 거래 대금을 예치하여 거래가 확정된 상태
- `거래완료` : 오프라인 직거래 현장에서 도서를 최종 확인한 후, 구매자가 **[구매 확정]**을 눌러 보관 중이던 결제 대금이 판매자에게 정산(송금)된 상태 (이후 평점 작성 플로우로 이동)

> **[📝 참고: 에스크로 운영 주체]**
> Polbook 서비스가 직접 금융을 다루고 대금을 보관하는(전자금융업 등록 필요) 자체 에스크로를 구축하는 것은 법적/보안적 리스크가 큽니다.
> 따라서 실제 결제와 대금 보관은 **외부 결제 대행사(PG사 - 예: 토스페이먼츠, NHN KCP, 아임포트 등)의 에스크로 결제 API**를 연동하여 처리합니다.
> Polbook 서버는 이 PG사로부터 결제 고유 번호(UID)와 상태값(`결제대기`, `결제완료/보관중`, `정산완료`, `취소됨`)만 전달받아 데이터베이스에 기록하고 관리합니다.

### 🚫 2.5 신고 및 제재 시스템
- **신고 대상:** 노쇼(채팅 후 잠수), 타 물품 판매(책이 아닌 물건), 사기 의심, 불쾌한 언행 등
- **처리 방식:**
  - 누적 신고 횟수(예: 3회 이상) 도달 시 경고 조치 및 일정 기간 서비스 이용(글쓰기/채팅) 정지
  - 관리자(Admin) 페이지를 두어 신고 접수 내역 관리

### ⭐ 2.6 매너 온도 / 평점 기능
- 당근마켓의 '매너온도'와 유사한 신뢰도 시스템.
- **거래 완료 시 동작:**
  - 거래가 '거래 완료'로 변경되면 구매자와 판매자 양측에 리뷰/별점 (1~5점) 작성 팝업 또는 알림 전송.
  - 남겨진 평점은 해당 유저의 프로필(닉네임 옆)에 평균치로 반영되어, 다른 학우들이 거래 상대의 신뢰도를 미리 파악할 수 있음.

## 3. 추천 기술 스택 (예시)
- **Frontend:** **React.js (웹)** / **React Native (모바일 앱)**
  - *선정 이유:* React 문법을 기반으로 웹과 앱(iOS/Android)을 동시에 효율적으로 개발할 수 있으며, 코드를 상당 부분 공유할 수 있습니다.
- **Backend:** **Spring Boot (Java/Kotlin)**
  - *선정 이유:* RESTful API 서버로 구축하여 React(웹)와 React Native(앱) 양쪽 플랫폼에서 모두 요청을 주고받을 수 있는 완벽한 호환성을 제공합니다.
- **Database:** **MySQL**
  - *선정 이유:* 회원, 게시글, 거래, 평점 등 데이터 간의 테이블 관계가 명확하므로 RDBMS인 MySQL이 매우 적합합니다.
- **Infrastructure:** **AWS (Amazon Web Services)**
  - `EC2`: Spring Boot 백엔드 서버 호스팅 (초기 t2.micro 또는 t3.micro 권장)
  - `RDS (MySQL)`: 관계형 데이터베이스 관리 및 자동 백업 지원 (db.t3.micro 권장)
  - `S3`: 유저가 업로드하는 중고 책 사진 파일(이미지) 저장용 스토리지

## 4. 데이터베이스(ERD) 모델링 초안
정확한 설계를 위해 아래 테이블들을 기반으로 구체적인 ERD 다이어그램을 다음 단계에서 작성할 예정입니다.

- `Users` (사용자: 학번, 이메일, 비밀번호, 닉네임, 평점평균)
- `Books` (게시글: ID, 작성자 학번, 책 제목, 카테고리, 학년, 학기, 강의명, 교수명, 가격, 책 상태, 필기 여부, 사진 첨부, 거래장소, 거래상태, 등록일)
- `Locations` (장소: 교내 거래 핫스팟 목록 관리)
- `Wishlists` (찜 목록: 사용자 학번, 찜한 책 ID, 찜한 날짜)
- `ChatRooms` (채팅방: 방 ID, 책 ID, 구매자 학번, 판매자 학번, 생성일)
- `ChatMessages` (메시지: 메시지 ID, 채팅방 ID, 보낸 사람 학번, 메시지 내용, 읽음 여부, 전송 시간)
- `Payments` (결제내역: 결제 UID, 구매자 학번, 연결된 책 ID, 결제 금액, 결제 수단, 상태[대기/보관중/환불/정산완료], 결제 일시)
- `Settlements` (정산내역: 정산 ID, 결제 UID, 판매자 학번, 정산 금액, 상태[대기/지급완료], 완료 일시)
- `Reviews` (평점: 평가자, 피평가자, 연결된 책 ID, 점수, 코멘트)
- `Reports` (신고내역: 신고자, 피신고자, 사유, 처리상태)
