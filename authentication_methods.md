# 인증 방식 비교: 토큰 기반 인증(JWT) vs 세션 기반 인증

웹/앱 서비스에서 사용자가 로그인되어 있음을 확인(인가)하기 위해 주로 사용하는 두 가지 핵심 방식인 **세션(Session)**과 **토큰(Token, JWT)** 기반 인증에 대해 설명합니다.

---

## 1. 세션 기반 인증 (Session-based Authentication)

전통적으로 웹 서비스에서 가장 많이 사용해 온 방식입니다. 사용자의 로그인 상태를 **서버의 메모리나 데이터베이스** 등에 저장해 두고 관리합니다.

*   **작동 방식:**
    1.  사용자가 로그인(ID/PW)을 요청합니다.
    2.  서버는 계정 정보를 확인한 후, 고유한 **세션 ID(Session ID)**를 생성하여 서버 메모리나 DB에 저장합니다.
    3.  서버는 이 세션 ID를 클라이언트 장치의 브라우저 쿠키(Cookie)에 담아 응답합니다.
    4.  이후 클라이언트는 매 요청마다 쿠키에 담긴 세션 ID를 서버로 보냅니다.
    5.  서버는 전달받은 세션 ID를 자신의 메모리/DB와 대조하여 유효한 사용자인지 판단합니다.

*   **장점:**
    *   서버가 로그인 상태를 완전히 통제할 수 있습니다. (예: 특정 유저 강제 로그아웃, 중복 로그인 방지 등 구현이 매우 쉬움)
    *   클라이언트(쿠키)에는 의미 없는 임의의 문자열(세션 ID)만 저장되므로 정보 유출 위험이 상대적으로 적습니다.
*   **단점 (Polbook에 덜 적합한 이유):**
    *   **서버 부하:** 접속자가 많아질수록 서버 메모리에 저장해야 할 세션 정보도 늘어납니다.
    *   **확장성 문제 (Stateful):** 서버를 여러 대(가령 서버 A, B, C)로 늘릴 경우(Scale-out), 세션 정보를 서버 간에 공유하거나(Redis 등 추가 저장소 필요) 특정 서버로만 요청을 고정해야 하는 복잡한 처리가 필요합니다.
    *   **플랫폼 호환성:** 모바일 앱(React Native) 등 웹 브라우저가 아닌 환경에서는 쿠키를 통한 세션 관리가 번거로울 수 있습니다.

---

## 2. 토큰 기반 인증 - JWT (Token-based Authentication)

최근 모바일 앱과 REST API 서버 구조에서 가장 널리 쓰이는 방식입니다. 사용자의 인증 정보를 **클라이언트가 직접 보관**하는 형태입니다. 주로 **JWT (JSON Web Token)**라는 형식을 사용합니다.

*   **작동 방식:**
    1.  사용자가 로그인(ID/PW)을 요청합니다.
    2.  서버는 계정 정보를 확인한 후, 해당 유저를 식별할 수 있는 정보(예: 유저 ID)와 만료 시간 등을 담아 암호화된 **JWT(토큰)**를 발급하여 클라이언트에게 전달합니다. (이때 서버는 토큰을 따로 저장/기억하지 않습니다.)
    3.  클라이언트는 이 토큰을 로컬 스토리지, 세션 스토리지, 혹은 안전한 쿠키 등에 보관합니다.
    4.  이후 클라이언트는 서버에 요청을 보낼 때마다 HTTP 헤더(Authorization)에 이 토큰을 항상 실어서 보냅니다.
    5.  서버는 토큰을 받아, 오직 자신이 가진 **비밀키(Secret Key)**로 정상적으로 발급된 토큰인지(위변조 및 만료 여부)만 **검증(디코딩)**한 후 요청을 처리합니다.

*   **장점 (Polbook에 적합한 이유):**
    *   **무상태성(Stateless) 및 서버 부하 감소:** 서버가 유저의 상태를 기억할 필요 없이, 넘어오는 토큰의 유효성만 검증하면 되므로 메모리 부하가 거의 없습니다.
    *   **뛰어난 확장성:** 서버를 여러 대로 늘려도 별도의 세션 공유 저장소(Redis 등) 없이 각 서버가 동일한 암호키만 가지고 있으면 토큰을 검증할 수 있습니다.
    *   **다양한 기기 지원:** 모바일 앱, 웹, 외부 API 등 형태를 가리지 않고 헤더에 토큰만 담아 보내면 되므로 플랫폼 간 호환성이 매우 뛰어납니다. (React, React Native 모두 사용하기 좋음)
*   **단점:**
    *   토큰이 탈취당하면 만료될 때까지 서버에서 강제로 차단하기가 어렵습니다. (이를 보완하기 위해 수명이 짧은 Access Token과 수명이 긴 Refresh Token을 혼합해서 사용하는 방식을 주로 취합니다.)
    *   토큰 자체에 유저 정보를 담고 있어 세션 ID보다 데이터 크기가 상대적으로 큽니다.

---

### 결론 요약
Polbook은 React(웹)와 React Native(모바일 앱)에서 동시에 자원을 요청해야 하므로, 플랫폼 환경에 구애받지 않고 서버 부담을 덜어주는 **토큰 기반(JWT) 인증 방식**을 도입하는 것이 최상의 선택입니다.
